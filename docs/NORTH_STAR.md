# 북극성: 이상한 씨앗상회와 에이전트 네이티브 게임 운영사

## 1. 왜 이 문서가 최상위인가

이 저장소는 하나의 게임만 만드는 곳이 아니다. 현재는 한 레포 안에 두 프로젝트가 함께 있지만, 장기적으로는 서로 분리될 수 있다.

1. **게임 프로젝트**: `이상한 씨앗상회`
2. **운영사 프로젝트**: 에이전트 네이티브 게임 개발사/운영사

따라서 모든 작업은 두 질문에 동시에 답해야 한다.

- 이 변경이 `이상한 씨앗상회`를 더 귀엽고, 더 수집하고 싶고, 더 다시 오고 싶은 게임으로 만드는가?
- 이 변경이 에이전트가 스스로 이슈를 만들고, 우선순위를 정하고, 구현하고, 검증하고, PR을 만들고, 실패를 복구하는 운영사를 더 현실적으로 만드는가?

둘 중 하나만 달성하면 충분하지 않다. 게임만 좋아지면 일반 게임 프로젝트이고, 자동화만 좋아지면 제품 없는 에이전트 데모다. 이 프로젝트의 정체성은 **에이전트가 스스로 더 재미있는 게임을 계속 만들어가는 살아있는 게임 스튜디오**다.

## 2. 북극성 A: 게임으로서의 이상한 씨앗상회

### 한 문장

**이상한 씨앗에서 태어난 이름 있는 식물 생명체들이 정원을 자동화하고, 주문을 처리하고, 탐험과 연구를 열어가며 플레이어가 매일 돌아와 더 키우고 싶어지는 브라우저 idle collection tycoon.**

### 핵심 약속

플레이어는 작은 온실에서 씨앗을 고르고, 심고, 톡톡 건드리며 성장시키고, 수확 순간에 이름 있는 생명체를 얻는다. 생명체는 단순 보상 아이콘이나 도감 카드가 아니라 정원 경제에 참여하는 actor여야 한다. 게임을 닫아도 생산과 주문 준비는 이어지고, 돌아오면 바로 수확, 납품, 업그레이드, 다음 생명체 목표가 기다려야 한다.

### 경쟁작 기준 Production Bar

`이상한 씨앗상회`의 품질 기준은 “기능이 동작하는 귀여운 수집 UI”가 아니다. P0.5 이후의 기준은 Egg, Inc. / Idle Miner Tycoon / Cell to Singularity 같은 idle 경쟁작에서 검증된 production loop를 이 게임의 fantasy에 맞게 축소 구현하는 것이다.

경쟁작에서 차용하는 핵심 기준:

1. **생산 엔진이 화면에 보여야 한다**: 정원은 정적인 카드 목록이 아니라 생명체가 일하고, 자원이 쌓이고, 주문 상자가 채워지는 공간이어야 한다.
2. **업그레이드 선택이 있어야 한다**: 다음 행동은 단순 `확인`이나 `열기`가 아니라 생산 속도, 밭, 주문, 탐험, 연구 중 무엇을 키울지의 선택으로 읽혀야 한다.
3. **다음 해금 목표가 계속 보여야 한다**: 플레이어는 다음 생명체, 다음 주문, 다음 시설, 다음 탐험, 다음 연구 중 적어도 하나를 항상 기대해야 한다.
4. **오프라인 진행이 의미 있어야 한다**: 돌아왔을 때 숫자만 늘어나는 것이 아니라 수확/납품/해금 순간이 기다려야 한다.
5. **장기 메타의 실루엣이 있어야 한다**: Phase 0라도 연구, 시즌/온실 확장, 탐험, 희귀 생명체 같은 장기 구조를 UI와 content hint로 보여줘야 한다.
6. **UI와 에셋은 core loop의 일부여야 한다**: 생명체 상태, 주문 상자, reward FX, 생산 tick, resource icon은 장식이 아니라 플레이어가 loop를 이해하는 gameplay surface다.

이 기준 때문에 `$seed-ops`의 다음 작업 선택은 작은 기능 checklist가 아니라 vertical slice 기준으로 판단한다. 즉, 한 작업은 가능하면 `player verb + production/progression role + screen moment + required assets/FX + playtest evidence`를 함께 개선해야 한다.

### 첫 번째 감정 목표

첫 5분 안에 플레이어가 이렇게 느껴야 한다.

> “얘 귀엽다. 하나만 더 키워볼까?”

P0.5부터는 이 감정 목표가 한 단계 확장된다.

> “이 아이가 정원을 실제로 움직이네. 다음 주문/업그레이드/생명체까지 보고 싶다.”

### Phase 0 핵심 루프

```text
씨앗 선택
-> 심기
-> 탭/기다림
-> 수확
-> 이름 있는 생명체 소유
-> 생명체가 정원 생산에 참여
-> 자동 생산/주문 납품
-> 업그레이드/새 씨앗/탐험/연구 해금
-> 더 희귀한 생명체 수집
-> 오프라인 복귀
```

작업 단위가 이 루프의 한 칸만 건드릴 수는 있다. 하지만 PR 설명과 issue plan은 반드시 그 칸이 전체 production loop의 어느 부분을 강화하는지 적어야 한다.

### 게임 북극성 지표

초기에는 실제 analytics보다 로컬 QA와 플레이테스트 기록으로 측정한다.

| 지표 | 의미 |
| --- | --- |
| 첫 씨앗 선택률 | 시작 즉시 무엇을 해야 하는지 이해했는가 |
| 첫 생명체 획득 시간 | 90초 안에 수집 보상이 오는가 |
| 첫 수확 감정 점수 | 수확 순간이 귀엽고 만족스러운가 |
| 도감 확인률 | 생명체가 재화가 아니라 소유 대상처럼 느껴지는가 |
| 두 번째 행동 전환율 | 첫 수확 후 다음 씨앗/밭/보상으로 이어지는가 |
| D1 복귀 의도 | 오프라인 보상과 수집 욕구가 재방문 이유가 되는가 |
| 생산 엔진 가시성 | 첫 화면에서 생명체/자원/주문 흐름이 보이는가 |
| 업그레이드 선택 명확도 | 다음 투자가 단순 버튼이 아니라 생산/수집/탐험 성장 선택으로 읽히는가 |
| 주문 납품 이해도 | 주문 목표, 진행률, 보상, 다음 해금이 한 흐름으로 이해되는가 |
| 게임 feel 반응성 | 탭, 수확, 납품, 보상 순간에 즉시 시각/수치 feedback이 오는가 |
| 장기 메타 기대감 | 연구/탐험/희귀 생명체/온실 확장 같은 다음 층이 보이는가 |

### 첫 5분 fun rubric

플레이테스트와 feedback intake는 아래 5개 축을 1-5점으로 기록한다. `first_5m_clarity`, `cuteness`, `collection_desire`, `comeback_hook`은 높을수록 좋고, `confusion`은 혼란도이므로 낮을수록 좋다.

| Rubric | 좋은 점수의 의미 | 1점 | 3점 | 5점 |
| --- | --- | --- | --- | --- |
| `first_5m_clarity` | 첫 5분 동안 무엇을 해야 하는지 바로 이해한다 | 시작 행동을 못 찾음 | 안내를 읽으면 진행 가능 | 안내 없이도 씨앗 선택→수확→도감 보상 흐름을 따라감 |
| `cuteness` | 생명체가 귀엽고 이름 있는 존재처럼 느껴진다 | 보상 아이콘처럼 보임 | 이름/이미지는 기억남 | 성격/인사말까지 기억하고 스크린샷을 남기고 싶음 |
| `collection_desire` | 하나 더 키워 다음 생명체를 보고 싶어진다 | 첫 수확 후 멈춤 | 다음 행동은 보이지만 욕구가 약함 | 도감/미발견 힌트 때문에 즉시 다음 씨앗을 심고 싶음 |
| `comeback_hook` | 게임을 닫았다가 다시 올 이유가 보인다 | 꺼도 손해/기대가 없음 | 오프라인 보상은 이해함 | 돌아오면 수확/보상/새 생명체가 기다린다고 기대함 |
| `production_readability` | 생산/주문/업그레이드 loop가 게임 화면으로 읽힌다 | 카드/설명만 있고 엔진이 안 보임 | 자동 생산은 보이나 다음 선택이 약함 | 생명체 actor, 자원 흐름, 주문/업그레이드 목표가 한 화면에서 이어짐 |
| `game_feel` | 조작과 보상이 즉시 반응한다 | 클릭해도 변화가 약함 | 수치/버튼 상태는 변함 | 탭/수확/납품 순간에 FX, motion, resource feedback이 즉시 보임 |
| `confusion` | 혼란도는 낮을수록 좋다 | 막힘 없이 진행 | 일부 CTA나 보상 의미가 애매함 | 심각한 혼란 또는 시작 행동을 찾지 못함 |

`confusion`은 역방향 지표다. 보고서에서는 점수를 그대로 기록하되, 우선순위 판단에서는 `confusion >= 4`를 high severity 후보로 본다.

### 게임 작업의 비타협 원칙

- 이름 있는 생명체 수집이 중심이다. 단순 숫자 증가만으로는 부족하다.
- 생명체는 도감 보상으로 끝나면 실패다. 최소 하나의 생산/주문/탐험/연구 역할을 가져야 한다.
- 첫 화면은 생산 엔진이 보여야 한다. 정적인 메뉴, 설명 패널, dashboard card 정리는 production quality가 아니다.
- 한 issue가 작은 UI를 바꾸더라도 `player verb`, `core loop role`, `screen moment`, `asset/FX need`, `playtest evidence`를 정의해야 한다.
- `viewport 안에 있음`, `겹치지 않음`, `DOM text visible`은 production bar가 아니라 최소 위생 조건이다.
- 첫 5분은 복잡한 시스템 설명보다 귀여운 결과와 다음 행동을 우선한다.
- Phase 0에서는 실제 결제, 실제 광고 SDK, 로그인/account, 외부 배포, 런타임 이미지 생성을 넣지 않는다.
- 상점은 mock/click-intent 검증까지만 허용한다.
- 수집 욕구는 귀여움과 소유감에서 나와야 하며, 숨겨진 확률/압박/다크패턴에서 나오면 안 된다.

## 3. 북극성 B: 에이전트 네이티브 게임 개발사/운영사

### 한 문장

**사람이 매번 지시하지 않아도, 에이전트가 게임 상태와 플레이어 신호를 읽고, 이슈를 만들고, 우선순위를 정하고, 구현하고, 검증하고, PR을 만들고, 실패를 복구하고, 운영 기록을 남기는 자율 게임 스튜디오.**

### 핵심 약속

사람은 CEO/크리에이티브 디렉터처럼 방향, 취향, 안전 경계, 승인 기준을 정한다. 에이전트는 운영팀처럼 매일 프로젝트를 읽고, 작은 단위의 개선을 만들고, 증거를 남기고, 실패하면 복구하거나 정직하게 중단한다.

### 첫 번째 운영 목표

처음부터 24시간 완전 자율 운영사를 만들 수는 없다. 가장 가까운 목표는 다음이다.

> `$ralph` 세션 안에서 가능한 오래, 가능한 많이, 가능한 안전하게 돌아가며 이슈 -> 작업 -> 검증 -> draft PR -> CI 확인 -> 후속 이슈까지 닫힌 루프로 수행한다.

### 운영사 북극성 지표

| 지표 | 의미 |
| --- | --- |
| 연속 자율 실행 시간 | 사람이 추가 지시하지 않아도 안전하게 계속 도는가 |
| heartbeat 신선도 | 봇이 살아있는지 외부에서 확인 가능한가 |
| issue-to-PR 완료 수 | 이슈 생성부터 draft PR까지 스스로 도달하는가 |
| CI green 복구율 | red PR을 끝 상태로 착각하지 않고 복구하는가 |
| stuck recovery 수 | `collab: Wait`, 멈춘 tmux, orphan process를 감지하고 회수하는가 |
| evidence completeness | 로그, 스크린샷, report, PR comment가 충분한가 |
| feedback-to-item 시간 | 고객/플레이테스트 신호가 work item으로 바뀌는가 |
| 안전 중단 정확도 | 결제, credential, 실제 배포, 개인정보, 브랜드 리스크에서 멈추는가 |

### 운영사 작업의 비타협 원칙

- PR이 red이면 완료가 아니다. 실패 원인, 복구 시도, 남은 blocker를 남겨야 한다.
- “오래 돌았다”는 말은 heartbeat와 로그로 증명해야 한다.
- main 자동 merge는 Branch protection, required checks, `ENABLE_AGENT_AUTOMERGE`, 명시 승인 전까지 금지한다.
- 고객 피드백, GTM, SNS, 이메일, 광고, 결제 계정은 실제 credential/브랜드/개인정보 리스크가 있으므로 mock intake와 승인 gate부터 만든다.
- 에이전트는 자율적이어야 하지만 무책임하면 안 된다. 모든 결정은 item, report, PR comment, dashboard 중 하나에 남아야 한다.

## 4. 현재 한 레포 안에서의 분리 원칙

현재 저장소는 게임과 운영사 실험이 함께 있다. 분리 전까지는 다음 경계를 사용한다.

| 영역 | 현재 위치 | 장기 분리 방향 |
| --- | --- | --- |
| 게임 제품 코드 | `src/`, `public/assets/`, `assets/source/` | `strange-seed-shop` 게임 repo |
| 게임 제품 문서 | `docs/PRD_PHASE0.md`, `docs/ECONOMY_PHASE0.md`, `docs/DESIGN_SYSTEM.md` | 게임 repo의 product/design docs |
| 운영사/에이전트 시스템 | `docs/AUTONOMOUS_PROJECT_OPERATING_MODEL.md`, `.omx/`, `scripts/check-*`, `reports/audits/` | 별도 agent-studio/operator repo 또는 package |
| 작업 기록 | `items/`, `reports/` | 양쪽 repo에 남기되 운영사 repo가 cross-repo dashboard를 소유 |

분리되기 전까지는 `docs/NORTH_STAR.md`가 양쪽의 공통 헌장 역할을 한다.

## 5. 단계별 마일스톤 원칙

운영사 프로젝트는 24시간 봇부터 만들지 않는다. 아래 순서로 올라간다.

1. **Ralph-session 운영사**: 한 Codex/OMX 세션 안에서 issue -> PR -> CI 확인을 반복한다.
2. **Watchdog 운영사**: stuck session, timeout, orphan process를 감지하고 재시작한다.
3. **Queue 운영사**: roadmap, items, GitHub issues를 읽고 다음 작업을 점수화한다.
4. **Feedback 운영사**: 플레이테스트/고객 피드백을 intake하고 중복/심각도/제품축으로 분류한다.
5. **GTM 운영사**: 승인된 범위 안에서 landing copy, devlog, changelog, release note, 커뮤니티 응답 초안을 만든다.
6. **24시간 운영사**: heartbeat, budget, safety gate, escalation을 갖춘 장시간 agent loop를 운영한다.
7. **분리형 운영사**: 게임 repo 여러 개를 고객처럼 다루는 별도 에이전트 네이티브 스튜디오가 된다.

각 단계는 그 자체로 검증 가능해야 하며, 다음 단계의 전제 조건을 남겨야 한다.

## 6. 우선순위 판단법

작업 후보가 여러 개일 때 다음 순서로 판단한다.

1. **게임의 첫 5분 재미를 직접 개선하는가?**
2. **경쟁작 기준 production loop를 강화하는가?** 생산 엔진, 업그레이드 선택, 주문/납품, 오프라인 복귀, 장기 메타 중 하나 이상을 화면과 gameplay에 연결해야 한다.
3. **수집 생명체의 귀여움/소유감/재방문 욕구를 강화하는가?**
4. **UI polish가 아니라 vertical slice를 진전시키는가?** `player verb + core loop role + screen moment + asset/FX + playtest evidence` 중 3개 이상이 없으면 우선순위를 낮춘다.
5. **에이전트가 더 오래, 더 안전하게, 더 검증 가능하게 일하게 하는가?**
6. **실패했을 때 복구 가능하고 기록이 남는가?**
7. **Phase 0 금지 영역을 건드리지 않는가?**

게임 재미와 운영사 자동화가 충돌하면, Phase 0에서는 게임 재미를 우선하되 운영사 자동화가 그 결정을 기록하고 검증하게 만든다.

### 작은 작업 선택 금지선

다음 유형의 작업은 단독으로는 높은 우선순위가 아니다.

- 색, 여백, 문구만 바꾸고 gameplay verb가 바뀌지 않는 작업
- 테스트가 `보인다/겹치지 않는다`만 확인하고 playtest finding이 없는 작업
- asset을 추가하지만 화면 순간이나 생명체 역할이 없는 작업
- 운영사 문서를 추가하지만 다음 게임 issue 선택 품질을 바꾸지 않는 작업

이런 작업은 production loop를 막는 blocker를 제거하거나, 명확한 vertical slice의 일부일 때만 선택한다.
