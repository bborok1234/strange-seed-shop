Status: draft
Scope-risk: moderate

## 요약

현재 Phase 0 앱은 첫 플레이 루프와 하단 탭 구조를 갖추고 있다. 다음 작업은 기능 확장이 아니라, 반복 작업자가 기존 루프를 깨지 않고 화면 밀도와 CTA 배치를 정리할 수 있도록 디자인 시스템과 검증 기준을 고정하는 것이다.

악마의 변호인 관점에서 현재 UI의 가장 큰 문제는 "작동은 하지만 게임의 첫 3초를 장악하지 못한다"는 점이다. 모바일 첫 화면에서 플레이어가 가장 먼저 읽어야 할 것은 지금 가능한 심기, 성장, 수확 행동인데, 현재는 잠긴 밭과 보조 패널이 비슷한 무게로 보인다.

`game-studio:game-ui-frontend` 기준으로도 현재 위험은 명확하다. 정원 화면은 일반 앱의 대시보드가 아니라 플레이필드여야 하는데, 현재 UI는 영구 패널과 3x3 잠긴 밭이 플레이 공간을 압박한다.

## 유지해야 할 루프

## Findings

### Critical: 모바일 첫 화면에서 잠긴 밭이 핵심 행동보다 먼저 보임

- 증거: Browser Use 관찰에서 모바일 폭 화면은 잠긴 밭이 큰 면적으로 노출되고, 씨앗/수확/업그레이드 행동은 아래 패널에 의존한다.
- 영향: 첫 20초 스타터 선택, 첫 90초 생명체 획득 목표를 방해한다.
- 조치 기준: 활성 밭과 현재 다음 행동 패널을 첫 viewport의 주인공으로 재배치한다. 잠긴 밭은 축소, 흐림, 접힘, 또는 진행 예고 영역으로 약화한다.

### High: 하단 탭은 작동하지만 현재 위치와 가치 제안이 약함

- 증거: `정원`, `씨앗`, `도감`, `원정`, `상점` 전환은 확인됐으나, 탭 간 정보 위계가 약하다.
- 영향: 사용자가 다음에 어디를 눌러야 하는지 UI가 돕지 못한다.
- 조치 기준: 활성 탭 indicator, 최소 터치 영역, 텍스트 대비 기준을 정의한다.

### High: 모바일 정원 탭이 스크롤 기반이 될 위험

- 증거: 현재 모바일 관찰에서 하단 탭 아래 보조 패널이 이어지고, 핵심 행동과 보조 정보가 한 화면 안에서 명확히 분리되지 않는다.
- 영향: 방치형/탭 게임의 핵심 루프가 웹페이지 탐색처럼 느껴진다. 정원 탭에서 스크롤을 요구하면 "작은 게임판" 감각이 깨진다.
- 조치 기준: 모바일 정원 탭은 360x800/360x900에서 핵심 루프를 세로 스크롤 없이 완료해야 한다. 스크롤은 씨앗, 도감, 상점 같은 보조 목록 탭 내부에만 허용한다.

### High: 상점 mock 카드가 미완성 화면처럼 보일 위험

- 증거: Browser Use 상점 탭 관찰에서 이미지가 깨져 보이고, 사용자에게 `mock` 텍스트가 노출된다.
- 영향: monetization intent 실험 품질이 낮아진다. 사용자는 관심 없음이 아니라 신뢰 부족으로 클릭하지 않을 수 있다.
- 조치 기준: `mock` 원문 노출 제거, "미리보기/관심 확인/실제 결제 없음" 카피 적용, 이미지 fallback과 정적 보강 에셋 후보 정의.

### Medium: 씨앗 구매/심기 액션의 우선순위가 동일하게 보임

- 증거: 구매/심기 버튼이 같은 row 안에서 비슷한 무게로 노출된다.
- 영향: 초반에는 지금 가능한 하나의 행동이 더 중요하다.
- 조치 기준: 상태별 primary action을 하나로 강조하고, 보조 행동은 secondary 스타일로 낮춘다.

### Medium: 디자인 토큰/컴포넌트 레이어 부족

- 증거: `src/styles.css`에 기능 단위 스타일이 누적되어 있고, 토큰/상태/컴포넌트 기준이 명확하지 않다.
- 영향: 이후 UI 수정이 화면별 임시 패치로 누적될 가능성이 높다.
- 조치 기준: 색상, 간격, 버튼, 카드, 탭, 밭 상태의 최소 토큰과 컴포넌트 계약을 문서화한다.

### Medium: 성장 컨셉에 맞는 모션 피드백이 부족함

- 증거: 씨앗 성장, 탭 가속, 수확 가능 상태가 대부분 정적인 UI 상태로 표현된다.
- 영향: 씨앗이 자라난다는 컨셉이 화면에서 충분히 살아나지 않는다. 유저가 탭으로 시간을 줄였다는 피드백도 약하다.
- 조치 기준: 성장 idle, 탭 반응, 수확 가능 상태, 보상 reveal에만 짧고 은은한 모션을 둔다. `prefers-reduced-motion`에서는 반복 모션을 줄인다.

## 계량 Acceptance Criteria

- 360px viewport에서 하단 탭 5개가 모두 보이고 각 탭 버튼 텍스트가 부모 밖으로 넘치지 않는다.
- 360x800/360x900 viewport에서 정원 탭 핵심 루프가 세로 스크롤 없이 조작 가능하다.
- 1280px viewport에서 하단 탭 5개가 모두 보이고 현재 탭 패널의 핵심 CTA가 화면 밖으로 밀리지 않는다.
- 정원 탭에서 첫 세션 루프가 Browser Use로 완료된다.
- 씨앗, 도감, 원정, 상점 탭 전환이 Browser Use DOM 또는 스크린샷으로 확인된다.
- 성장 중, 탭 반응, 수확 가능 상태, 보상 reveal에만 제한적 motion이 적용되고 reduced-motion 기준이 문서화된다.
- 상점 CTA 클릭 후 analytics log에는 `shop_surface_clicked`만 추가된다.
- 상점 CTA 클릭으로 URL 변경, 새 창, 결제 SDK, 로그인/계정 UI가 발생하지 않는다.
- `npm run check:all` 통과 또는 실패 시 실패 원인과 후속 item이 기록된다.

## 선택지와 결정

### Option A: 문서만 작성하고 UI는 그대로 둔다

위험은 낮지만 Milestone 3.5의 실제 시각 QA 개선 수용 기준을 충족하지 못할 수 있다. 지금 문제는 문서 부재만이 아니라 첫 화면의 정보 구조와 상점 mock의 신뢰도 문제다.

### Option B: 디자인 시스템 문서 + 제한된 CSS/표면 UI 정리 + Browser Use/CDP 재검증

첫 5분 UX와 토큰/컴포넌트 계약을 함께 정리한다. 변경 범위는 moderate지만, 금지 변경 조건을 명시하면 기존 루프를 보존하면서 문제를 해결할 수 있다.

### Option C: UI 컴포넌트 구조 전면 재편

장기적으로는 더 깔끔하지만 현재 작동 루프와 QA 증거를 흔들 가능성이 크다. Phase 0 안정화 목적에 비해 범위가 과하다.

Decision: Option B.

## 잔여 위험

- 현재 앱은 `src/App.tsx`에 UI와 게임 로직이 함께 있어 표시-only 변경과 로직 변경의 경계가 흐릴 수 있다.
- 모바일 360px 기준은 캡처만으로 모든 동적 텍스트 overflow를 보장하지 못하므로 DOM 스냅샷 또는 수동 관찰을 함께 남겨야 한다.
- 디자인 시스템 초안은 사용 규칙 중심이며, 컴포넌트 라이브러리화는 별도 milestone로 분리해야 한다.
